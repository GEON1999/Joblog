# 지원 상태는 단계×결과의 직교 조합

지원(Application)의 상태를 하나의 enum이 아니라 진행 단계(Stage: 지원함→서류→과제→면접→오퍼)와 결과(Outcome: 진행중/탈락/철회/수락)의 직교 조합으로 모델링한다. 탈락을 칸반 컬럼(단계의 하나)으로 두면 "어느 단계에서 떨어졌는지"가 소실되어 전환율 대시보드가 성립하지 않기 때문이다. "서류에서 탈락"은 Stage=서류 × Outcome=탈락으로 남는다.

## Considered Options

- **탈락을 칸반 컬럼으로 추가** — 구현은 단순하지만 탈락 시점 단계가 사라지고, 시간이 지나면 탈락 컬럼만 비대해진다.
- **이벤트 소싱** — 모든 전환을 이벤트로 저장하고 상태를 파생. 싱글유저 CRUD 앱에는 과잉이다.

## Consequences

- 종료된 지원은 칸반에서 아카이브로 빠지되, 마지막 Stage 정보를 보존한다.
- 체류 일수 계산을 위해 단계 전환 이력(Stage Transition) 테이블은 별도로 둔다. 이는 이벤트 소싱이 아니라 단순 전환 로그다.
- 전환율의 분모는 결판난 건(다음 단계로 넘어간 건 + 그 단계에서 종료된 건)만 포함한다. 대기 중인 건을 분모에 넣으면 통과율이 체계적으로 낮게 왜곡된다.
