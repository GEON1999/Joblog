# 제출 문서 파일은 private 버킷 + 서버 전용 service role + 서명 URL

이력서 등 제출 문서 파일은 Supabase Storage의 **private 버킷**에 저장하고, 서버에서만 접근한다. 업로드와 서명 URL 발급에는 **service role 키**를 쓰되 이 키는 절대 클라이언트로 노출하지 않는다(`NEXT_PUBLIC_` 접두사 없이 `SUPABASE_SERVICE_ROLE_KEY`). 다운로드는 로그인 세션을 확인한 뒤 짧은 수명의 서명 URL을 발급해 리다이렉트한다. ADR 0002의 "파일은 private, 서버 경유 접근" 방침을 파일 계층에서 구체화한 것이다.

## Considered Options

- **public 버킷** — URL만 알면 누구나 접근. 이력서는 개인정보이므로 불가.
- **anon 키 + Storage 정책(RLS)** — 정책을 세밀히 짜면 되지만, DB에서 이미 "Data API를 잠그고 서버 전용 경로로 몬다"(ADR 0005)는 모델을 택했다. 스토리지도 같은 모델로 통일해 접근 판단을 애플리케이션 서버 한 곳에 두는 것이 일관적이다.
- **파일을 서버로 스트리밍 다운로드** — 서버가 대역폭을 떠안는다. 서명 URL은 발급만 서버가 하고 전송은 Supabase가 처리해 더 가볍다.

## Consequences

- service role 키는 RLS·스토리지 정책을 **전부 우회**하는 강력한 키다. 서버 모듈에만 import하고(`server-only` 가드), 클라이언트 번들·로그·에러 메시지에 새지 않게 한다. 유출 시 Supabase 대시보드에서 로테이션한다.
- 버킷(`documents`)은 배포 전에 대시보드에서 private으로 생성해야 한다(README에 안내). 앱은 버킷 존재를 전제한다.
- DB의 문서 레코드와 스토리지 파일은 별개 저장소라 원자적이지 않다. 삭제·업로드 실패 시 고아 레코드/파일이 남을 수 있어, 삭제는 파일 → 레코드 순으로 처리하고 업로드는 파일 성공 후 레코드를 만든다.

## Update (2026-07-24)

다운로드는 처음에 **짧은 수명의 서명 URL로 리다이렉트**하는 방식이었으나, Supabase의 서명 URL `download` 옵션이 한글 등 유니코드 파일명을 **이중 인코딩**해 다운로드 파일명이 깨지는 문제가 있었다. 그래서 **서버가 파일을 직접 스트리밍**하고 RFC 5987(`filename*=UTF-8''…`)로 Content-Disposition을 만드는 방식으로 바꿨다. 대역폭을 서버가 지는 트레이드오프가 생기지만, 싱글유저·소용량(≤4MB)·간헐적 다운로드라 무시할 수준이고, 공개 URL을 아예 만들지 않아 접근 제어가 오히려 더 단순·견고해졌다. 서명 URL은 더 이상 쓰지 않는다.
