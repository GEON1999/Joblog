# 개인 도구에서 공개 가입 오픈 서비스로 — 앱 레벨 멀티테넌시

JobLog를 1인용 개인 도구에서 **누구나 공개 가입 가능한 오픈 서비스**로 전환한다. 채용 담당자가 직접 회원가입해 자기 데이터만 격리해서 체험하는 것이 목적이다. 데이터 격리는 **MVP에서 앱 레벨 필터링**으로 구현한다 — 루트 테이블에 `userId`를 두고 모든 쿼리가 현재 유저로 필터링한다. DB 레벨 RLS(`auth.uid()`)는 지금 하지 않고 향후 하드닝으로 남긴다. ADR 0002의 "개인용, 회원가입 없음" 전제를 정면으로 뒤집는 결정이라 별도 ADR로 기록한다.

## Context

기존 구조는 회원가입이 없는 것이 미구현이 아니라 의도된 설계였다(ADR 0002). 그 결과 (1) 모든 데이터 테이블에 `user_id`가 없고, (2) Drizzle이 `DATABASE_URL` 직접 연결로 RLS를 우회하며(ADR 0005), (3) 쿼리에 유저 필터가 하나도 없다. 따라서 오픈 서비스 전환의 핵심은 가입 폼이 아니라 **데이터 소유권 모델 도입**이다.

## Considered Options

### 격리를 어디서 강제하는가 — 앱 레벨 vs DB 레벨 RLS

- **앱 레벨 필터링(채택)** — 루트 테이블에 `userId`를 추가하고 모든 쿼리/액션이 `WHERE userId = 현재유저`로 필터. 지금의 직접 연결 구조를 유지한다. 방어선이 앱 코드 하나뿐이라 필터를 한 곳만 빠뜨려도 유출된다는 리스크가 있어, `userId`를 쿼리 함수의 **명시적 인자**로 전달해 시그니처에서 격리 여부가 드러나게 하고, 셀프리뷰로 보강한다.
- **DB 레벨 RLS(`auth.uid()`)** — 트랜잭션마다 유저 JWT/claims를 실은 연결로 바꿔 RLS 정책이 DB에서 차단. 심층 방어지만 Drizzle 직접 연결 구조를 재설계해야 한다(`SET LOCAL role`/`request.jwt.claims` 래퍼, 접속 role이 `authenticated`로 전환 가능해야 함). MVP 규모 대비 비용이 커 **향후 하드닝**으로 미룬다. 지금도 RLS는 "켜짐 + 정책 없음(=거부)"이라 anon/authenticated 경로는 이미 막혀 있어, 직접 연결만 쓰는 한 앱 레벨로도 안전하다.

### `userId`를 어느 테이블까지 두는가

- **루트만(채택)** — `companies`·`applications`·`documents`에만 `userId`. 자식(`stage_transitions`·`interviews`·`offers` 등)은 부모 조인으로 소유권 파생. denormalize의 부모-자식 불일치(=유출) 리스크를 피하고 백필이 루트 3개로 끝난다. 자식을 id로 직접 수정/삭제하는 액션은 부모까지 조인해 소유권을 검증한다.
- **전 테이블 denormalize** — 어떤 테이블이든 단독 필터 가능하지만 정합성 부담이 크고 앱 레벨에선 불일치 버그가 곧 유출이다.

### 접근제어 — 화이트리스트를 어떻게 하는가

- **오너 등급 목록으로 전환(채택)** — `AUTH_ALLOWED_EMAILS`를 삭제하지 않고 "로그인 관문"에서 "무제한 오너 등급"으로 의미를 바꾼다. 목록에 있으면 오너(모든 상한 면제), 없으면 일반 공개 유저(쿼터 적용). 새 테이블/`role` 컬럼 없이 등급을 표현한다. `isAllowedEmail` → `isOwner`.
- **완전 삭제** — 누구나 가입/사용은 되지만, 프로젝트 소유자 본인이 쿼터에 걸린다. 오너 예외 수단이 사라진다.

### 가입 마찰 — 이메일 인증

- **인증 없이 즉시 로그인(채택)** — 대부분의 채용 담당자는 확인 메일을 귀찮아해 이탈한다. 마찰을 0으로 둔다. 대신 이메일 인증이라는 1차 방벽이 사라지므로 남용 방어를 캡차로 대체한다.
- **이메일 인증 필수** — 봇/오타 이메일을 거르지만 `auth/callback`·커스텀 SMTP가 필요하고 담당자 마찰이 커진다.

### 남용/비용 방어

- **캡차(Turnstile) + 스토리지 총량 쿼터(채택)** — Supabase Auth 네이티브 Turnstile로 봇 대량가입 차단(인비저블이라 마찰 낮음), 유일한 실질 비용 축인 스토리지에 파일당 4MB(기존) + 비오너 유저당 총량 상한. IP 레이트리밋·행 개수 상한은 향후로 남긴다.

### 유저별 ICS 피드 토큰

- **HMAC 파생 토큰(채택)** — `HMAC(ICS_FEED_SECRET, userId)`. `?uid=&token=`로 폴링, 서버가 상수시간 비교 후 `userId`로 필터. 새 테이블 없이 유저별 격리. 개별 revoke는 전역 시크릿 회전으로만 가능(데모 규모에서 수용).
- **유저별 토큰 DB 저장** — 개별 revoke가 쉽지만 "새 테이블 없음" 기조를 깬다.

### 신규 유저 온보딩

- **빈 상태 + 첫 방문 코치마크 + 샘플 데이터 토글(채택)** — 강제 시딩 대신 유저가 "샘플 채우기/비우기"로 완성된 보드·차트를 직접 확인. 개인정보 노출 없이 앱 가치를 보여준다. 코치마크 완료 플래그는 localStorage.

## Consequences

- **격리 정확성이 앱 코드에 달려 있다.** `userId` 필터 누락은 곧 데이터 유출이다. 쿼리 함수 시그니처의 명시적 `userId`, 소유권 검증 헬퍼, 셀프리뷰 체크리스트로 방어한다.
- **기존 본인 데이터는 백필된다.** 전 루트 행의 `userId`를 오너 계정 uid로 채운다(`nullable 추가 → 백필 → NOT NULL`). 담당자 계정과 격리되어 본인 데이터는 본인만 본다.
- **`companies.name` 전역 유니크가 깨진다.** 두 유저가 같은 회사명을 못 쓰는 문제를 피하려 `unique(userId, name)` 복합 유니크로 바꾼다.
- **service role 스토리지는 여전히 앱이 격리를 책임진다.** 다운로드 서명/스트리밍 전 `document.userId === 현재유저`를 반드시 검증한다. 경로를 `{userId}/{documentId}/{원본파일명}`로 네임스페이싱해 정리·감사를 쉽게 하고, 향후 스토리지 RLS 전환의 발판을 만든다.
- **캡차는 로그인·가입 모든 auth 엔드포인트에 적용된다.** Supabase에서 CAPTCHA를 켜면 오너 로그인 폼에도 `captchaToken`이 필요하다(인비저블이라 마찰 미미).
- **신규 운영 준비물**: Supabase 대시보드에서 Confirm email OFF, Allow signups ON, CAPTCHA(Turnstile) 활성화. 신규 env `ICS_FEED_SECRET`, `TURNSTILE_SECRET_KEY`, `NEXT_PUBLIC_TURNSTILE_SITE_KEY`. `AUTH_ALLOWED_EMAILS`는 오너 목록으로 존치.

## 향후 개선

- **DB 레벨 RLS 하드닝** — `auth.uid()` 정책 + 트랜잭션 래퍼로 격리를 DB 최종 방어선까지 내린다. 앱 레벨 필터가 심층 방어의 1차선이 된다.
- IP 레이트리밋, 유저당 행 개수 상한, ICS 개별 토큰 revoke(토큰 버전/유저 설정 테이블 승격).
